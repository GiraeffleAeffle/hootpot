Hootpot Architecture

A production-shaped map for HOOT membership, group funding, merchant receipts, and random cashback paybacks.

Circles group CRC receipts Gnosis Pay sync Safe payout pot

The short answer

The HOOT Circles group is enough for joining and community support. The extra Hootpot contracts are not needed for a user to join HOOT or mint/support HOOT. They are optional proof and operator contracts for merchant allowlists, receipt registration, draw records, and non-CRC prize payouts.

Current blocker

The HOOT open service is deployed, but the group owner Safe still has to set it as the group service once. Until then, each member has to be accepted by the group owner/service before that user can mint HOOT.

System map

1. Join HOOT

User trusts HOOT. If open join is enabled, the service also makes HOOT trust the user. Both directions are needed for a clean group mint path.

2. Support HOOT

User sends CRC to the HOOT mint handler. The group treasury receives backing collateral and the user receives HOOT group tokens.

3. Create receipts

Receipts come from Circles merchant checkout or from Gnosis Pay transaction metadata through SIWE sync or signed webhooks.

4. Pay cashback

Hootpot selects one eligible receipt and pays the user back from the payout Safe or a future treasury sweep.

HOOT group

  • A Identity and community funding layer.
  • B Members are accounts the group trusts.
  • C Affiliate starring is recurring passive support.

Hootpot app

  • A Builds Circles transactions through the miniapp host.
  • B Stores receipts, draw state, and payout proofs.
  • C Fetches group, pot, and user state from Gnosis Chain.

Optional contracts

  • A Open group service makes membership self-serve.
  • B Receipt registry publishes round and draw proofs.
  • C Prize pool is useful for non-CRC prize custody.

User flow in the app

  1. 1 Open Hootpot inside the Circles host or playground.
  2. 2 Click Join HOOT. This builds the trust and open-service transactions.
  3. 3 After indexing, enter an amount and click Mint HOOT.
  4. 4 Pay a participating merchant or sync Gnosis Pay receipts.
  5. 5 Eligible receipts enter the cashback draw.

Operator flow

  1. 1 Deploy HootpotOpenGroupService with the HOOT group address.
  2. 2 Set NEXT_PUBLIC_HOOTPOT_GROUP_OPEN_SERVICE_ADDRESS and redeploy.
  3. 3 Open ?operator=1 as the HOOT owner Safe and click Enable Open Join.
  4. 4 Fund the payout Safe, run a draw, send the payback, and record the payout transaction.

Why people had to be trusted manually

In Circles protocol groups, group members are the accounts the group trusts. For a base group, the service address exists specifically so owners can automate membership updates. With no service set, the owner Safe is the practical path.

Why Gnosis Pay still matters

Gnosis Pay can provide real card receipt metadata, but production domains need SIWE/CORS whitelisting and webhooks must be configured by the partner setup. Until then, Circles checkout is the directly demonstrable payment path.

What is production-ready vs still missing

Implemented now

  • HOOT group state lookup from Gnosis Chain.
  • Join HOOT transaction builder.
  • HOOT mint/fund transaction builder through the mint handler.
  • Owner-only open join setup transaction builder.
  • Gnosis Pay SIWE sync and webhook ingestion endpoints.
  • Receipt ledger, draw state, payout recording, and pot balance.

Still required externally

  • Set the deployed open group service as the HOOT service.
  • Configure real merchant payout addresses.
  • Configure durable Postgres or KV storage in Vercel.
  • Get hootpot.vercel.app whitelisted by Gnosis Pay.
  • Fund the payout Safe or wire a treasury sweep policy.
  • Replace blockhash draw randomness with a stronger source.

Reference facts from the docs

  • Circles groups have members defined as accounts the group trusts; member CRC can be minted 1:1 into group CRC.
  • Base groups have an owner, service address, fee collection address, treasury, mint handler, and membership conditions.
  • The base mint handler lets users mint group tokens by sending CRC to the handler, using pathfinding and wrapped balances.
  • Affiliate groups are selected by starring a group page; the selected affiliate group receives 1/12 of daily CRC issuance.
  • Gnosis Pay production domains must be registered for SIWE/CORS, and webhooks need a public HTTPS endpoint.
